# 01 — 业务背景与算法决策

> 这份文档回答两个问题:**我们要做什么?** 和 **为什么这么做(以及为什么不那么做)?**
> 是 [`chat.md`](../chat.md) 中讨论的提炼与立场归档。AI 协作时**先读这份**,再读执行类文档。

---

## 1. 业务问题定义

### 1.1 场景
工厂/电商需要对印有 Logo 的服装、海报、包装等进行 **Logo 一致性质检**:

- **正常变形(必须容忍)**:布料褶皱、上身拉伸、光影折射、轻微旋转、视角偏移
- **异常缺陷(必须识别)**:
  - **错版**:Logo 印错版本(配色、字形、整体结构错误)
  - **漏印**:Logo 部分缺失(缺笔画、缺角、缺色块)
  - **污染**:墨点、印刷糊版、色彩串位
  - **几何错位**:Logo 镜像、倒置、错位粘贴

### 1.2 输入 / 输出

| | 内容 |
|---|---|
| **输入** | (a) 一张"标准 Logo 参考图"<br>(b) 一张"待检图像"(已经过 Logo ROI 裁剪,或带定位框) |
| **输出** | (a) 一致性得分 ∈ [0, 1]<br>(b) 二分类决策(pass / fail)<br>(c) 进阶:异常区域热力图(定位缺陷位置) |

### 1.3 业务约束
- **零样本起步**:Logo 种类多、新款上线快,**不能要求每个 Logo 都标注几百张异常样本**
- **吞吐**:生产线上单张推理 < 100ms(5090 上 DINOv3-Large 已满足)
- **可解释性**:质检员需要看到"为什么判 fail",仅有标量分数不够 → 需要热力图

## 2. 技术选型:为什么是 DINOv3

| 方案 | 优势 | 劣势 | 结论 |
|---|---|---|---|
| 传统 SIFT / ORB 关键点匹配 | 经典、可解释 | 对柔性形变(布料)鲁棒性差;光照敏感 | ❌ 不适合 |
| 监督学习专用模型(ResNet 分类) | 简单 | 需要大量标注;Logo 多样性下泛化差 | ❌ 数据成本高 |
| CLIP 图文对齐 | 语义特征强 | 训练目标偏向"语义类别"而非"实例细节",对错版/漏印这种细粒度差异不敏感 | ⚠️ 备选 |
| **DINOv3 自监督视觉特征** | **Patch 级密集特征**;**对几何形变天然鲁棒**;**无需标注** | 仅是特征提取器,需要在其上设计相似度度量 | ✅ **采用** |

DINOv3 的核心价值是 **它学的是"视觉对象本身"而非"类别标签"**,Patch token 携带丰富的局部纹理/结构信息,正好匹配 Logo "哪里像、哪里不像" 这种细粒度比对需求。

## 3. 度量方案:GAP vs Patch-EMD vs Deformable Attention

这是 [`chat.md`](../chat.md) 中**最关键的争议点**,本项目的立场归档:

### 3.1 方案 A:全局平均池化 + 余弦相似度(Gemini 推荐)

**做法**:
```
DINOv3 features [B, N, D]  →  mean over N  →  [B, D]  →  L2 normalize  →  cosine similarity
```

**优势**:
- 实现极简(5 行代码)
- 推理快、向量小、可建索引
- 对褶皱、轻微形变天然鲁棒(平均化抹平局部偏移)

**致命盲区**(Gemini 自己也承认):
> GAP 抹平了所有空间信息——它"忽略"褶皱的同时,也**彻底抹平了 Logo 的局部微小缺陷**。
> Logo 边缘缺一个角、字母少一笔、墨点污染——GAP 后的全局向量几乎不变。
> 在工业质检中**极易引发高漏检率**。

**适用边界**:
- ✅ 粗粒度场景:整体错版(配色完全不同)
- ❌ 细粒度场景:漏印、缺笔、墨点——这正是质检的主要场景

### 3.2 方案 B:Patch 级最优传输(Patch-EMD / Optimal Transport)

**做法**:
```
DINOv3 features  →  保留 Patch grid [B, H*W, D]  (例如 14×14 = 196 个 patch)
                →  对参考图 patch 集 R 与待检图 patch 集 Q
                →  计算 Earth Mover's Distance (EMD) / Sinkhorn-OT 距离
```

**核心思想**:不要求 patch 一一对齐(允许褶皱导致的空间漂移),但要求"集合层面"必须能完整匹配——**任何缺失的 patch 都会被惩罚**。

**优势**:
- 容忍形变(空间漂移在最优传输下被吸收)
- **能捕捉局部缺失**(EMD 对集合差异敏感)
- 可生成 patch 级匹配代价矩阵 → **天然产出热力图**

**代价**:
- 计算量:N×N 距离矩阵 + Sinkhorn 迭代,单对比对 ~10-50ms(5090 上仍可接受)
- 实现复杂度高于 GAP

### 3.3 方案 C:Deformable Attention 微调

**做法**:在 DINOv3 之后挂一个轻量的 Deformable Attention 头,学习"针对褶皱纹理动态预测采样偏移",显式做软对齐再比对。

**评估**:
- 上限最高,但需要微调数据(LoRA 也行)
- Phase 3 的探索方向,**不作为 MVP**

### 3.4 本项目立场

> **Phase 1 用方案 A 跑通基线**——不是因为它最好,而是因为它**最快验证整条流水线(数据 → DINOv3 → 度量 → 决策)是否畅通**。
>
> **Phase 2 切换到方案 B(Patch-EMD)** 作为生产路线——这才是匹配业务场景的正确度量。
>
> **Phase 3 评估方案 C** 作为精度上限攻坚。

详细路线见 [`05-roadmap.md`](./05-roadmap.md)。

## 4. Gemini 对话中其他需要警惕的盲区

| Gemini 论点 | 本项目立场 |
|---|---|
| Triplet Loss 微调 | ⚠️ Hard Negative 挖掘对 Logo 边界样本极敏感,容易模式崩溃。**Phase 3 优先用 InfoNCE / SupCon,Triplet 作为消融对照**。 |
| 弹性形变数据增强(TPS、仿射) | ⚠️ 算法模拟形变 ≠ 真实布料形变。**优先采集真实穿着照片**,合成增强仅作 warmup。 |
| `torch.compile(mode="reduce-overhead")` | ✅ 推荐,但首版先跑 eager 模式确认正确性,Phase 2 再开 compile。 |
| `bitsandbytes` 4-bit 加载 7B | ✅ 必须。但首版用 Large 即可,7B 是 Phase 3 攻坚才用。 |
| `AutoImageProcessor` 默认 Resize 到正方形 | 🚨 **海报是长方形,直接 Resize 会破坏几何结构**。**必须**改用 aspect-ratio preserving padding(OpenCV 实现)。这是 P0 的坑,见 `06-pitfalls.md`。 |
| DINOv3 7B 全量微调要多卡 A100 | ❌ 误导。LoRA + 4bit 量化下单卡 5090 完全够用。 |
| `--exclude "*.msgpack" "*.h5"` 下载剪枝 | ✅ 推荐,可省几 GB 磁盘 |

## 5. 数据策略(待定,Phase 1 完成后决策)

## 6. 评估指标

- **二分类**:Precision / Recall / F1,**优先关注 Recall(漏检代价 >> 误检)**
- **排序**:mAP@K(给定一张待检图,标准库中 top-K 是否命中正确 Logo)
- **定位**:异常热力图 IoU(若有像素级标注)
